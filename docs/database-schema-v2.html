Architecture Overview

This schema implements a single-tenant, multi-outlet, multi-terminal POS. The cloud DB is the source of truth for master data; each terminal keeps a local SQLite database (encrypted with SQLCipher) and syncs both directions.

Key Design Decisions (locked in)

  • IDs: Hybrid — every table has Integer PK (fast joins) plus uuid column (cross-DB identity, sync-safe).
  • Menu: Global master products + outlet_products override table per outlet.
  • Staff: Bound to one outlet; can log in on any terminal in that outlet. PIN for cashiers, password for managers.
  • Tables: Stored at outlet level with optional assigned_terminal_id.
  • Stock: Per-outlet count + immutable stock_movements log for audit.
  • VAT: Sequential gap-less tax_invoice_no per outlet, allocated centrally via invoice_counters with row-locking.
  • Payments: Separate payments table — supports split payments (cash + card on same order).
  • Audit: Single audit_log capturing auth events & sensitive actions (voids, refunds, price changes).
  • Encryption: SQLCipher on local DB (AES-256). PostgreSQL uses TLS in transit + at-rest encryption (cloud provider).
  • Customers: Optional table for future loyalty — nullable FK on orders.

Sync Model (manual + 10-min auto)

Each terminal works fully offline on its local SQLite DB. Sync to cloud happens via two operations (manual buttons + scheduled at sync_interval_minutes):

  • SYNC MASTER (cloud → terminal): Pulls categories, products, outlet_products, tables, staff, settings — filtered by the terminal's outlet_id. Only rows with updated_at > last_master_sync_at are pulled.
  • SYNC TRANSACTIONS (terminal → cloud): Pushes orders, order_items, payments, stock_movements, audit_log entries with sync_status = 'pending'. On success cloud returns the assigned tax_invoice_no and the row's sync_status flips to synced.

Every synced row carries outlet_uuid, terminal_uuid, local_uuid, updated_at — the cloud uses the UUID as the deduplication key (idempotent push).

Security Design

🔐 Local DB Encryption

SQLite file encrypted with SQLCipher (AES-256). Encryption key derived from device UUID + PBKDF2-stretched secret stored in OS keychain (Windows Credential Manager / macOS Keychain / Android Keystore).

🔑 Tiered Authentication

Cashiers: 6-digit PIN, bcrypt cost-12. Managers/Admins: username + strong password, bcrypt cost-12. Sensitive actions (voids, refunds, discounts >X%) require manager password re-entry.

🚦 Rate Limiting

5 failed PIN attempts in 5min → 15min lockout per staff. 10 failed terminal-API attempts → terminal blacklisted, requires admin re-activation.

🪪 Token Architecture

Two layers: (1) Terminal token — long-lived, signed with per-terminal secret, identifies the device. (2) Staff JWT — short-lived (4h), refreshed on activity, identifies the user.

📋 Audit Trail

Every sensitive action logged with: who (staff_uuid), when (UTC), where (terminal_uuid, ip), what (action_type), details (JSONB old/new). Audit log is append-only — no deletes from app code.

🛡️ Data Isolation

Cloud queries always filtered by outlet_id from the terminal's JWT. Staff cannot query other outlets even by ID guessing. Backend enforces scope at every endpoint.

Hybrid ID Strategy — Integer PK + UUID

Every synchronized table has both:

  • id — Integer auto-increment, PRIMARY KEY (fast joins inside a single DB).
  • uuid — UUID v4 (or v7 for time-ordered), UNIQUE NOT NULL, generated at the terminal at insert time. This is the sync identity.

Why both?

Integer PK stays small (4 bytes), B-tree indexes are dense and fast for joins within one database.

UUID guarantees no collisions when Terminal-A and Terminal-B both create rows offline. The cloud uses uuid as the ON CONFLICT key, so re-pushing the same row is idempotent.

Foreign keys between tables in the same database use the integer id. Sync messages between databases reference rows by uuid.

CLOUD DATABASE — PostgreSQL (Source of Truth)
PostgreSQL is the source of truth for all master data and the long-term store for transactions. All FK relationships are enforced. Every table has id (BIGSERIAL) and uuid (UUID, indexed UNIQUE). Timestamps are TIMESTAMPTZ. Money columns use NUMERIC(14,2) instead of FLOAT for exact arithmetic.

LOCAL DATABASE — SQLite (per terminal, SQLCipher-encrypted)
SQLite mirrors the cloud schema for shared tables, plus terminal-only tables (settings, sync_settings, table_statuses, etc.). Each terminal has its own DB file, encrypted with SQLCipher. Tables that sync to cloud carry: uuid, sync_status, synced_at, updated_at. Foreign keys between local tables use Integer IDs; cross-DB references use UUIDs.
